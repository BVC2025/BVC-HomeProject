# Module 02 — Human Resources

The HR module is the largest cluster in BVC24. It covers five sub-modules that share the `employee` table as their anchor:

1. **Employee Master** — directory, profile, photos, resume fields.
2. **Attendance** — daily check-in / check-out with biometric integration.
3. **Leave Management** — applications with one-click email approvals.
4. **Payroll** — monthly computed slips from attendance + tasks.
5. **STAR Performance** — monthly star ratings with promotion recommendations.

## 2.1 Employee Master

### Screens

- **Employees** (`/employees`) — directory with search, department filter, employee cards with photos / avatars / skill tags.
- **EmployeeProfileForm** (`/employee-profile/...`) — first-login self-registration gate (six sections: personal, contact, education, professional, family, notes).
- **AddEmployeeModal** — admin-side employee creation modal (6-section form).
- **ResumeModal** — full-profile view styled as a printable résumé, with the **sticky-header** modal pattern (header fixed, body scrolls).

### Workflow — onboarding a new employee

```
1. Admin POSTs /create-employee with code, name, email, phone,
   department, designation, role, joining date.
2. The backend hashes the default password (initial) with bcrypt
   and stores it.
3. Admin shares EMPLOYEE_CODE + initial password with the employee.
4. Employee logs in via POST /employee-login → lands on
   EmployeeProfileForm (because PROFILE_SUBMITTED=0).
5. Employee fills the 6-section form and submits via
   POST /employees/by-code/{code}/submit-profile.
6. Backend sets PROFILE_SUBMITTED=1 — the form is locked and
   becomes read-only on subsequent logins.
7. Employee is now on the standard EmployeeDashboard.
```

### Key API endpoints

| Endpoint | Purpose |
|---|---|
| `POST /create-employee` | Admin creates employee |
| `GET /employees` | Directory list |
| `GET /employees/{id}` | Profile detail with resolved dept/designation/role |
| `GET /employees/by-code/{code}` | Lookup by `EMPLOYEE_CODE` |
| `POST /employees/by-code/{code}/submit-profile` | Self-registration |
| `PUT /update-employee/{id}` | Edit |
| `PUT /employees/{id}/reset-password` | Admin password reset |
| `POST /employees/{id}/upload-photo` | Profile picture |
| `DELETE /delete-employee/{id}` | Remove |

### Data

`employee` table — see [Schema §6.2](../06-database-schema.md#62-people). Key behaviour:

- `STATUS` — `ACTIVE` / `SUSPENDED` / `RESIGNED` / `TERMINATED`. Suspended/Resigned employees are excluded from task allocation and attendance-active counts but their records persist.
- `FINGERPRINT_ID` — links to biometric device USER_ID, allowing automated attendance from raw scans.
- `PROFILE_SUBMITTED` — gate flag; controls whether the self-registration form is read-only.
- `SKILLS` — comma-separated tags used by the workload allocator and the Gantt auto-assignment heuristic.

## 2.2 Attendance

### Screens

- **Attendance** (`/attendance`) — admin board with day/month/week views, present/late/absent badges, manual mark-absent.
- **BiometricCheckIn** (`/biometric`, no auth) — tablet kiosk screen for shop-floor check-in.

### Workflow — biometric check-in

```
1. Employee places finger on the device.
2. Device firmware HTTP-POSTs to /biometric/scan with
   { device_id, fingerprint_id, event_time, verify_mode, result }.
3. Backend creates a BiometricEvent row (raw audit log).
4. If fingerprint_id matches employee.FINGERPRINT_ID:
   - First scan of the day → Attendance CHECK_IN at event_time.
   - Subsequent scan → Attendance CHECK_OUT.
   - STATUS computed from CHECK_IN time vs. SHIFT_START:
       on/before  → PRESENT
       after by < 30 min  → LATE
       no scan today      → ABSENT (set by daily cron)
5. WORKED_HOURS = CHECK_OUT − CHECK_IN (rounded).
```

### Manual override

- `POST /attendance/check-in`, `POST /attendance/check-out`, `POST /attendance/mark-absent` allow admin to record attendance without a biometric event.

### Key endpoints

`GET /attendance`, `GET /attendance/today`, `GET /attendance/live-board`, `DELETE /attendance/{id}`.

## 2.3 Leave Management

### Screens

- **ApplyLeave** (`/apply-leave`) — employee-facing form (no auth required if linked via deep link with employee code, otherwise normal login).
- **LeaveManagement** (`/leave-management`) — admin board with pending, approved, rejected, balance views.

### Workflow — token-based email approval

```
1. Employee submits via POST /leave/apply
       { LEAVE_TYPE: CASUAL, START_DATE, END_DATE, REASON }
2. Backend validates against LeaveBalance quota:
       - Insufficient balance with EARNED → rejected
       - Insufficient balance with CASUAL/SICK → auto-convert to LOP
                                                 (loss of pay)
3. LeaveRequest row created with:
       STATUS = PENDING_APPROVAL
       APPROVAL_TOKEN = secrets.token_urlsafe(24)
4. Email dispatched to APPROVER_EMAIL with two single-click links:
       <BACKEND_URL>/leave/decide/<token>?action=approve
       <BACKEND_URL>/leave/decide/<token>?action=reject
5. Approver clicks the link from their phone/desktop — no portal
   login required. Backend renders an HTML decision page,
   sets APPROVED_BY_EMAIL, updates STATUS, and writes
   APPROVAL_RESOLVED_AT.
6. Employee receives a confirmation notification.
7. LeaveBalance.*_USED is updated atomically with the approval.
```

### Leave types & quotas (defaults)

| Type | Default quota / year | Notes |
|---|---|---|
| `CASUAL` | 12 days | Auto-converts to LOP if quota exhausted |
| `SICK` | 12 days | Same as casual |
| `EARNED` | 15 days | Rejected if quota exhausted (cannot auto-convert) |
| `UNPAID` / `LOP` | unlimited | Always approved (subject to admin override) |

### Key endpoints

`POST /leave/apply`, `GET /leave/decide/{token}?action=`, `PATCH /leave/{id}/approve|reject`, `GET /leave/pending`, `GET /leave/my-requests`, `PATCH /leave/{id}/cancel`, `GET /leave/balance/{employee_id}`.

## 2.4 Payroll

### Screens

- **Payroll** (`/payroll`) — admin board with monthly runs, individual slips, finalize / mark-paid actions.

### Workflow — generate a payroll run

```
1. Admin POSTs /payroll/generate { PAY_YEAR, PAY_MONTH }
2. Backend (payroll_service.py):
   - Fetches all ACTIVE employees for the vendor
   - For each employee:
     - Counts attendance rows in the month:
         days_present, days_late, days_half, paid_leave, unpaid_leave
     - Computes PER_DAY_RATE = BASE_SALARY / WORKING_DAYS (26)
     - Counts COMPLETED tasks for the month → TASK_BONUS = N × 100
     - Calculates LATE_PENALTY (configurable per vendor)
     - Computes GROSS_PAY, TOTAL_DEDUCTIONS, NET_PAY
   - Creates PayrollRun + N PayrollSlip rows
3. Admin reviews each slip on the UI
4. PATCH /payroll/runs/{id}/finalize  → STATUS = FINALIZED (locked)
5. PATCH /payroll/runs/{id}/mark-paid → STATUS = PAID
```

### Salary formula (per-employee, per-month)

```
PER_DAY_RATE   = BASE_SALARY / WORKING_DAYS
EARNED_BASIC   = PER_DAY_RATE × (DAYS_PRESENT + 0.5×DAYS_HALF + PAID_LEAVE_DAYS)
TASK_BONUS     = TASKS_COMPLETED × TASK_BONUS_PER_TASK (default 100)
OT_PAY         = OT_HOURS × (PER_DAY_RATE / 8) × 1.5
GROSS_PAY      = EARNED_BASIC + TASK_BONUS + OT_PAY
DEDUCTIONS     = LATE_PENALTY + OTHER_DEDUCTIONS
NET_PAY        = GROSS_PAY − DEDUCTIONS
```

### Data

`payroll_run`, `payroll_slip` — see [Schema §6.10](../06-database-schema.md#610-payroll--performance).

## 2.5 STAR Performance

### Screens

- **StarPerformance** (`/star-performance`) — leaderboard with monthly star ratings, top performers, promotion / increment recommendations.
- **MDReview** (`/md-review`) — MD-only view with action buttons to approve / deny recommendations.

### Scoring formula (monthly, per employee)

```
ATTENDANCE_STARS    = (DAYS_PRESENT / WORKING_DAYS) × 5
TASK_STARS          = (TASKS_ON_TIME / TASKS_ASSIGNED) × 5
PRODUCTIVITY_STARS  = (ESTIMATED_HOURS / max(ACTUAL_HOURS, 1)) × 5
CONSISTENCY_STARS   = derived from variance of daily completion

OVERALL_STARS = 0.25 × ATTENDANCE_STARS
              + 0.30 × TASK_STARS
              + 0.25 × PRODUCTIVITY_STARS
              + 0.20 × CONSISTENCY_STARS
```

Thresholds (configurable):

- `OVERALL_STARS >= 4.5` → `RECOMMENDED_FOR_PROMOTION = 1`
- `OVERALL_STARS >= 4.0` → `RECOMMENDED_FOR_INCREMENT = 1`

### Workflow

```
1. POST /performance/stars/compute   (end-of-month cron or manual)
   - Recomputes PerformanceScore for every employee for the month
2. GET /performance/stars/top        → leaderboard
3. PATCH /performance/stars/{id}/action  → MD approves/denies the
                                           recommendation; on
                                           approval, REWARDED = 1
                                           and a notification is sent
```

### Data

`performance_score` — see [Schema §6.10](../06-database-schema.md#610-payroll--performance).

## 2.6 Cross-cutting Patterns

- **Sticky-header modals** are used for all employee detail views — the resume modal and the AddEmployeeModal both fix the title bar so it stays visible while scrolling a long form.
- **Skills as tags** — `Employee.SKILLS` is comma-separated; the allocator searches for skill keywords (e.g. `welding`, `electrical`) when picking the least-loaded matching employee.
- **Audit timeline** — performance recommendations write to the activity log when approved/denied.
